# スキル設計パターン 5選

Anthropic公式ガイドで紹介されている、実績のある5つのパターン。

---

## 問題ファーストvs.ツールファーストの考え方

- **問題ファースト**: 「プロジェクトをセットアップしたい」→ スキルが適切なMCP呼び出しを順番に調整する。ユーザーはゴールを伝えるだけ。
- **ツールファースト**: 「Notion MCPを接続している」→ スキルが最適なワークフローとベストプラクティスを教える。アクセス手段はある、スキルが専門知識を提供する。

---

## パターン1: Sequential Workflow（順次ワークフロー）

**使いどき**: ユーザーが特定の順序で多段階プロセスを必要とする場合。

**構造例**:
```markdown
## ワークフロー: 新規顧客オンボーディング

### Step 1: アカウント作成
MCPツール呼び出し: `create_customer`
パラメータ: name, email, company

### Step 2: 支払い設定
MCPツール呼び出し: `setup_payment_method`
待機条件: 支払い方法の検証完了

### Step 3: サブスクリプション作成
MCPツール呼び出し: `create_subscription`
パラメータ: plan_id, customer_id（Step 1 から取得）

### Step 4: ウェルカムメール送信
MCPツール呼び出し: `send_email`
テンプレート: welcome_email_template
```

**主な技術**:
- ステップの明示的な順序付け
- ステップ間の依存関係の定義
- 各段階でのバリデーション
- 失敗時のロールバック手順

---

## パターン2: Multi-MCP Coordination（複数MCP連携）

**使いどき**: ワークフローが複数のサービスにまたがる場合。

**構造例**:
```markdown
### Phase 1: デザインエクスポート（Figma MCP）
1. Figmaからデザインアセットをエクスポート
2. デザイン仕様書を生成
3. アセットマニフェストを作成

### Phase 2: アセット保存（Drive MCP）
1. Driveにプロジェクトフォルダを作成
2. すべてのアセットをアップロード
3. 共有リンクを生成

### Phase 3: タスク作成（Linear MCP）
1. 開発タスクを作成
2. アセットリンクをタスクに添付
3. エンジニアリングチームに割り当て

### Phase 4: 通知（Slack MCP）
1. #engineering にハンドオフサマリーを投稿
2. アセットリンクとタスク参照を含める
```

**主な技術**:
- フェーズの明確な分離
- MCP間のデータ引き渡し
- 次フェーズに進む前のバリデーション
- 集中型エラーハンドリング

---

## パターン3: Iterative Refinement（反復改善）

**使いどき**: 反復によって出力品質が向上する場合。

**構造例**:
```markdown
## 反復レポート生成

### 初稿作成
1. MCP経由でデータを取得
2. 最初のドラフトを生成
3. 一時ファイルに保存

### 品質チェック
1. 検証スクリプトを実行: `scripts/check_report.py`
2. 問題を特定:
   - 欠落しているセクション
   - 不一致なフォーマット
   - データ検証エラー

### 改善ループ
1. 各問題に対処する
2. 影響を受けたセクションを再生成
3. 再検証する
4. 品質閾値に達するまで繰り返す

### 最終化
1. 最終フォーマットを適用
2. サマリーを生成
3. 最終版を保存
```

**主な技術**:
- 明示的な品質基準の定義
- 反復改善ループ
- 検証スクリプトの活用
- 終了条件の明示

---

## パターン4: Context-Aware Tool Selection（状況対応ツール選択）

**使いどき**: 同じゴールに対してコンテキストに応じて異なるツールを使う場合。

**構造例**:
```markdown
## スマートファイル保存

### 判断ツリー
1. ファイルの種類とサイズを確認する
2. 最適な保存先を判断する:
   - 大きいファイル（10MB超）: クラウドストレージMCP
   - 共同編集ドキュメント: Notion/Docs MCP
   - コードファイル: GitHub MCP
   - 一時ファイル: ローカルストレージ

### 保存を実行する
判断に基づいて:
- 適切なMCPツールを呼び出す
- サービス固有のメタデータを適用する
- アクセスリンクを生成する

### ユーザーへの説明
その保存先を選んだ理由を説明する
```

**主な技術**:
- 明確な判断基準
- フォールバックオプション
- 選択に対する透明性

---

## パターン5: Domain-Specific Intelligence（ドメイン特化知識）

**使いどき**: スキルがツールアクセスを超えた専門知識を追加する場合。

**構造例**:
```markdown
## コンプライアンス対応の支払い処理

### 処理前（コンプライアンスチェック）
1. MCP経由でトランザクション詳細を取得
2. コンプライアンスルールを適用する:
   - 制裁リストを確認する
   - 管轄の許可を検証する
   - リスクレベルを評価する
3. コンプライアンス判断を文書化する

### 処理
コンプライアンス通過の場合:
- 支払い処理MCPツールを呼び出す
- 適切な不正検知チェックを適用する
- トランザクションを処理する
失敗の場合:
- レビュー用にフラグを立てる
- コンプライアンスケースを作成する

### 監査証跡
- すべてのコンプライアンスチェックを記録する
- 処理判断を記録する
- 監査レポートを生成する
```

**主な技術**:
- ドメイン専門知識をロジックに埋め込む
- アクションの前にコンプライアンスチェック
- 包括的なドキュメント化
- 明確なガバナンス

---

## パターン選択ガイド

| 状況 | 推奨パターン |
|------|------------|
| ユーザーが手順を知らなくてもゴールを達成したい | Sequential Workflow |
| 複数のSaaSサービスを連携させたい | Multi-MCP Coordination |
| ドキュメントやコードの品質を上げたい | Iterative Refinement |
| ファイル種別や状況で処理を分岐したい | Context-Aware Tool Selection |
| 法規制・業界ルールを遵守したい | Domain-Specific Intelligence |
